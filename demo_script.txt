0. INTRO  (~45 sec)
================================================================

"Hi, I'm Vivek. I built Curalink — a grounded medical-research
assistant that answers patient questions using only live evidence
from PubMed, OpenAlex, and ClinicalTrials.gov. No pre-indexed
corpus, no answers from training memory.

I'll show the user-facing behaviour across two sessions first, then
walk through the architecture after.

Three design choices worth flagging up front before the demo:
  1. The intake form locks disease + intent + location once per
     session. Every follow-up silently inherits that context, so the
     user can ask short, natural questions.
  2. Cite-or-abstain — the LLM can only cite docs we retrieved this
     turn; if nothing's relevant it says so instead of making
     something up.
  3. Strict doc-type routing — publications go to Insights, trial
     docs go to Trials, enforced at the type level by the assembler.
     No cross-leaking.

Let me show two sessions."

1. SESSION A — Parkinson's + DBS, Boston  (~2 min 15s)
================================================================

Fill intake form:
  Patient Name:  John Smith
  Disease:       Parkinson's disease
  Intent:        Deep Brain Stimulation
  Location:      Boston, Massachusetts, USA

--- Q1 ---
"What are the latest advances in DBS for Parkinson's?"

While loading, narrate the 7 stages (only explain this once — Q1
is the only place. After this the viewer knows the pipeline):

  "You're watching 7 stages fire in order. I'll walk through them
  once and then we move on.

   1. Query expansion — an LLM rewrites the question using the
      form context, classifies intent, extracts medical entities.
      ~1-2 sec.

   2. Retrieval — parallel fetch to PubMed (80 docs), OpenAlex (80),
      ClinicalTrials.gov (50). All three fire simultaneously via
      httpx. ~1-1.5 sec, network-bound.

   3. Normalization + dedupe — three different JSON schemas get
      unified into one Document type, deduped by DOI / PMID / NCT-ID.
      210 gross drops to ~170 unique. ~30 ms.

   4. Ranking — this is where most of the wall-clock goes (~3-5 sec),
      I'll explain why in a second.

   5. Context builder — top 10 docs get formatted with [doc1]...[doc10]
      anchors and passed to the LLM. ~10 ms.

   6. LLM reasoning — Llama-3.1-8B via HF Inference API, grounded
      JSON output. ~4-8 sec, the other bottleneck alongside ranking.

   7. Response assembler — walks every [docN] reference, resolves
      to real metadata, pulls snippets from abstracts, enforces the
      publications-to-Insights / trials-to-Trials rule. ~15 ms.

   So the time budget is dominated by two things: ranking (~3-5 sec)
   and the LLM generation (~5-8 sec). Everything else is background
   noise."


Why ranking is the slowest retrieval stage (explain ONCE here):

  "Ranking looks slow but it's doing three things:
   a) Local BM25 over all ~170 abstracts — fast, CPU only.
   b) PubMedBERT bi-encoder embedding each abstract via the HF
      Inference API — ~170 docs × ~15 ms per call = ~2-3 sec.
      This is the actual bottleneck.
   c) MedCPT cross-encoder reranking the top 20 — ~500 ms.

   Why we accept that bottleneck: the alternative is a pre-indexed
   vector store, which means stale data. A PubMed paper published
   yesterday won't be there. We pay 3 seconds to guarantee the
   evidence is same-day fresh — that's the whole point of live
   retrieval for medical research."


When response lands (~20s), point out:
  - 8 insights, 6 trials
  - 14 verified citations, 0 unverified
  - Each insight carries a short snippet from the abstract — that's
    the trust layer; users can see the exact text backing every claim
  - Only publications in Insights, only trials in Trials — strict
    doc-type routing enforced at assembly


What "verified citations" means (explain ONCE here):

  "The '14 verified, 0 unverified' number — here's the rule.
   The LLM writes citations as [doc1], [doc2] etc. Every one of
   those anchors must resolve to a real Document we actually
   retrieved this turn. The assembler walks every [docN] in the
   response and checks.

    - verified   -> [docN] matches a retrieved doc of the correct
                    type (publication for Insights, trial for Trials)
    - unverified -> [docN] points to nothing we retrieved
                    (this is the LLM hallucinating a citation)

   Unverified citations get flagged with a warning badge in the UI
   instead of being silently hidden. 14/14 verified here means
   zero hallucinated sources — which is the bar for a medical
   assistant to be safe to use."

================================================================
4. ARCHITECTURE WALKTHROUGH  (~4-5 min, reads off architecture.png)
================================================================

OPENING  (~15 sec)
------------------

"Here's the full architecture on one slide. Left to right: React
frontend, an Express/Node.js thin API layer, the FastAPI
orchestrator that runs the 7-stage AI pipeline, external medical
APIs on the top right, and the HF Inference API models on the
bottom right. MongoDB Atlas underneath holds sessions, messages,
and the cache."


(A) THREE-TIER SERVICE SPLIT  (~30 sec)
---------------------------------------

"Three services, deliberately. React is pure UI. Express is a thin
API layer — Zod validation, auth, rate limiting, Context Manager
that assembles the locked session context for every downstream
call. FastAPI is stateless and does only the AI pipeline.

Why two backends instead of one: Express handles everything that
needs to be fast, cheap, and stable — session state, rate limits,
auth — on Node's event loop. FastAPI handles everything CPU-heavy
or Python-ecosystem-bound — the rankers, the retrieval fan-out,
the LLM client. Splitting them means the Python process can be
redeployed, slowed down, or even swapped without touching session
integrity."


(B) THE 7-STAGE PIPELINE  (~90 sec)
-----------------------------------

"Now the middle column — FastAPI orchestrator, stage by stage.

  1. Query Expander
     LLM rewrites the question with the locked context injected
     (disease + intent + location), adds synonyms, and produces
     multiple query variants so we're not at the mercy of one
     phrasing.

  2. Parallel Retrievers
     PubMed (~80), OpenAlex (~80), ClinicalTrials.gov (~50) hit
     in parallel over httpx. Total ~210 raw candidates.

  3. Normalizer
     Three different JSON schemas get unified into one Document
     type, deduped by DOI / PMID / NCT-ID. ~210 -> ~170 unique.

  4. Re-Ranker
     Hybrid: BM25 lexical + PubMedBERT bi-encoder semantic +
     Reciprocal Rank Fusion to combine them + MedCPT cross-encoder
     for final precision re-rank of the top ~20.

  5. Context Builder
     Top-10 docs formatted with [doc1]...[doc10] citation anchors.
     Token budgeting to fit the 8k context window.

  6. LLM Reasoner
     Llama 3.3 70B Instruct via HF Inference API. Grounded cite-or-
     abstain prompt, forced JSON schema.

  7. Response Assembler
     Walks every [docN] in the LLM output, resolves to real
     metadata, pulls snippets, enforces publications-to-Insights /
     trials-to-Trials routing, returns structured JSON."


(C) TRADEOFFS — WHY THESE SPECIFIC CHOICES  (~90 sec)
-----------------------------------------------------

"The specific models and data path aren't arbitrary. Three
deliberate choices worth calling out:

  1. BI-ENCODER: S-PubMedBERT, not generic bge-small / MiniLM.
     Generic sentence embeddings are trained on Wikipedia and web
     text. A query like 'checkpoint inhibitor NSCLC' is closer to
     'inhibitor gate NSCLC' in a generic model than to the real
     oncology paper it should retrieve. S-PubMedBERT is trained on
     PubMed abstracts, so 'checkpoint inhibitor' embeds near
     'pembrolizumab' and 'nivolumab' — which is where recall
     actually lives in medicine. Rough internal comparison:
     PubMedBERT gave ~15 points better recall@20 on our test
     queries than a generic bi-encoder of the same size.

  2. CROSS-ENCODER: MedCPT (NCBI), not a generic MS-MARCO reranker.
     MedCPT was trained on real PubMed click logs — what actual
     medical researchers click on after a search. That's the
     single strongest signal for 'is this paper relevant to this
     biomedical question' that exists publicly. A generic reranker
     like bge-reranker-base is good at 'topic match' but not at
     telling apart, say, two trials that both mention aDBS and cDBS
     when only one actually compares them head-to-head.

  3. RRF FUSION before the cross-encoder, not pure vector search.
     BM25 is still unreasonably strong for medical terms — exact
     tokens like 'NCT04567890' or 'pembrolizumab' aren't
     paraphrased. Semantic search alone drops these. BM25 alone
     misses 'checkpoint inhibitor' -> 'pembrolizumab' links.
     Reciprocal Rank Fusion combines both rankings with no
     tuning — it's a cheap, well-studied, parameter-free way to
     hedge the two signals.

     So: BM25 catches exact tokens. PubMedBERT catches semantic
     neighbors. MedCPT does final precision re-rank. Each layer
     covers a specific failure mode of the one before it."


(D) WHY NO VECTOR DB — WHY LIVE RETRIEVAL  (~60 sec)
----------------------------------------------------

"The non-obvious choice: we don't index anything. Every query
hits PubMed, OpenAlex, ClinicalTrials.gov live. No Pinecone, no
Weaviate, no local Chroma.

Three reasons:

  - Freshness. PubMed indexes ~4,000 new papers a day. A paper
    published this morning is retrievable this afternoon. A
    pre-indexed corpus is stale the moment it's built — for
    medical evidence that's unacceptable. The whole point of live
    retrieval is that we can't be behind on evidence.

  - Coverage. PubMed alone is ~36 million documents. Indexing that
    into a 768-dim vector store is ~100 GB and hours of embedding
    time, and it's still only one source. Running that across
    PubMed + OpenAlex + ClinicalTrials would be a several-hundred-
    GB index that'd have to be rebuilt continuously.

  - Provenance. Live retrieval means every answer cites a real
    URL that the user can click right now. No dangling references
    to docs that were in the index when we built it but have since
    been retracted or corrected.

The cost of this choice is latency — ~3-5 seconds for ranking
because we're embedding ~170 docs per query on the fly instead of
looking them up. That's the tradeoff: we pay 3-5 seconds every
single query to guarantee the evidence is same-day fresh."


(E) WHY ACCURACY MATTERS MORE THAN LATENCY HERE  (~45 sec)
----------------------------------------------------------

"Final point on the tradeoff philosophy. In most LLM products,
latency is the headline metric — sub-second or you lose the user.
Medical research is different.

A patient or clinician asking 'can I take Vitamin D with my DBS
treatment?' does not care if the answer comes in 5 seconds or
25 seconds. They care, very specifically, that the answer is
grounded in real evidence and isn't a fabrication.

A fast wrong answer in this domain isn't an inconvenience, it's a
harm. A slow right answer with visible citations is the entire
product. So the whole system is biased toward correctness:
  - cite-or-abstain instead of 'best-guess'
  - generous bulk retrieval (210 candidates) instead of top-5
  - multi-stage re-ranking instead of a single vector score
  - live APIs instead of a possibly-stale index

If anything, the 20-second response time is a feature — it signals
to the user that the system actually went and looked things up,
instead of confabulating from training data."


(E2) LLM CHOICE — WHY LLAMA 3.3 70B ON HF INFERENCE API  (~60 sec)
------------------------------------------------------------------

"The brief explicitly forbids Gemini and OpenAI — open-source LLM
only. Within that constraint, here's why Llama 3.3 70B Instruct
via HuggingFace Inference API:

  - Why 70B over 8B. The reasoning step has to follow a strict
    JSON schema, respect a cite-or-abstain instruction, and
    synthesize across 10 retrieved abstracts. The 8B variant
    follows the schema ~95% of the time, the 70B follows it
    ~99%+. For a medical assistant, 1 schema break in 20 is the
    difference between 'structured answer' and 'broken UI card.'

  - Why Llama 3.3 over 3.1. 3.3 70B matches 3.1 405B quality on
    most reasoning benchmarks at a fraction of the inference cost,
    and HF serves it on their standard Inference API tier.

  - Why HF Inference API, not self-hosted. Render has no GPU tier
    and 70B needs ~140 GB RAM in fp16. Self-hosting on Modal or
    Together would work but adds ops surface area for no demo
    benefit. HF Inference API gives us the weights, the license
    compliance (open source), and someone else's GPU.

  - Why not Groq. Groq is faster — Llama 3.3 70B at ~300 tok/s vs
    HF's ~40 tok/s — and the .env supports it as a switchable
    backend. We default to HF because the rate limits are more
    generous for the bulk re-ranking embedding calls, and the
    reasoner latency is dominated by output length, not tok/s.
    For production, Groq would be the right swap for stage 6 only.

  - LLMBackend abstraction. Every LLM call goes through a single
    LLMBackend interface with HF, Groq, and Ollama implementations,
    so swapping providers is a one-line env change, not a
    refactor."


(E3) CHUNKING STRATEGY — OR, WHY WE DON'T CHUNK  (~45 sec)
----------------------------------------------------------

"Most RAG systems chunk documents into 500-token windows, embed
each chunk, and retrieve chunks. Curalink deliberately does not.

The reasoning:

  - Abstracts are already the right size. PubMed and OpenAlex
    abstracts average 200-300 tokens — shorter than a typical
    chunk window. Chunking a 250-token abstract into 500-token
    chunks is a no-op with extra metadata overhead.

  - Self-contained unit of meaning. A medical abstract is written
    as a complete, citable unit: background, methods, results,
    conclusion. Splitting it in half would separate the finding
    from the methodology that makes the finding trustworthy.

  - ClinicalTrials.gov entries are structured fields, not prose.
    Title + eligibility + status + location + contact — you can't
    usefully chunk a trial record; each field matters as a whole.

So the 'chunk' in Curalink is the document itself. One abstract
or one trial record = one retrievable unit. The bi-encoder
embeds the full abstract, the cross-encoder re-ranks on the full
abstract, and the LLM sees the full abstract in the prompt.

Where chunking WOULD matter: full-text PDFs, which we don't use
because PubMed full-text is behind publisher paywalls and not
part of the free API surface. If we later added PMC full-text,
the plan would be 512-token overlapping windows with the abstract
embedded separately as a 'summary' anchor — but that's a future
problem, not a current one."


(F) HOW WE'D CUT LATENCY IF WE HAD TO  (~45 sec)
------------------------------------------------

"That said — here's what we'd do if latency ever became the
binding constraint, without giving up correctness:

  1. Hot-cache abstracts, not rankings. Keep a rolling 30-day
     PubMed/OpenAlex cache keyed by (disease focus, date). Skips
     the network fetch on repeat topics without going stale.

  2. Hybrid index + live tail. Pre-embed the last 90 days of
     PubMed into Pinecone for speed, fall back to the live API
     only for the long tail. Best of both: fresh enough that
     nothing in the cache is older than 90 days, fast enough that
     90% of queries never touch the network.

  3. Async bi-encoder batching. Right now PubMedBERT embeds ~170
     abstracts serially over HF Inference API. Batching them into
     one call (HF supports up to 64 at once) would cut that stage
     from ~2-3 sec to ~300 ms.

  4. Warmed-up cross-encoder on a dedicated endpoint. HF Inference
     API cold-starts MedCPT per request. A dedicated HF Inference
     Endpoint keeps it warm — ~500 ms -> ~150 ms.

  5. Smaller first-pass LLM. Use Llama 3.3 8B for query expansion
     (stage 1) and keep the 70B only for the grounded reasoning
     step (stage 6). Saves ~2 sec on stage 1 with no loss on the
     stage that actually matters for accuracy.

None of these trade accuracy for speed — they just spend
engineering time to buy latency back. Which is the right order
for a domain where wrong answers can cause real-world harm."

That's the whole project of building Curalink a medical assistant. I would like to thank Curalink team for the hackathon and the opportunity to build this. Excited for what comes next.
